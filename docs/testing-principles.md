# テスト原則

## FIRST原則

| 原則 | 説明 |
|------|------|
| **F**ast | テストは高速に実行できること |
| **I**ndependent | テスト同士が独立し、順序に依存しないこと |
| **R**epeatable | 何度実行しても同じ結果になること |
| **S**elf-validating | 合否が自動判定されること（目視不要） |
| **T**imely | コードと同時期に書くこと |

## Right-BICEP（テスト観点）

| 観点 | 説明 |
|------|------|
| **Right** | 正常系が正しく動くか |
| **B**oundary | 境界値（0, 最大値, 閾値の前後） |
| **I**nverse | 逆操作で元に戻るか |
| **C**ross-check | 別の方法で同じ結果を検証 |
| **E**rror | エラー条件で安全に動くか |
| **P**erformance | 性能が許容範囲内か |

## テスト設計技法

- **同値分割**: 入力を等価クラスに分け、各クラスの代表値でテスト
- **境界値分析**: クラス境界の値（境界上・直前・直後）を重点テスト
- **状態遷移テスト**: 状態遷移図に基づき、有効・無効な遷移パスを網羅

## テストピラミッド

```
     /  E2E  \        ← 少数・遅い・高コスト
    / Integration \    ← 中間
   /    Unit Tests  \  ← 多数・速い・低コスト
```

- **Unit**: 純粋関数・ロジックの検証（vitest）
- **E2E**: ユーザー操作→画面反映のクリティカルパス（Playwright）
- E2Eは「ユニットでは検証できないUI統合のみ」に絞る

## 良いテストの判断基準

- 1テスト1目的
- 失敗時に原因が即特定できる
- **実装ではなく振る舞い**を検証する
- テストが壊れるのは「バグが入ったとき」だけ（実装変更では壊れない）
- 価値 = バグ検出力 / 保守コスト
